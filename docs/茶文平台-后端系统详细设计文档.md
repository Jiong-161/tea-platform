## 1. 编写目的与设计范围

### 1.1 编写目的

本文档对"茶文·茶文化品鉴购销服务平台"后端系统进行详细设计，明确各微服务模块的内部结构、数据库表结构、接口定义、安全策略、Sentinel 流控熔断策略和关键业务流程的实现方案，作为编码实现和后续维护的技术依据。

### 1.2 设计范围

本文档覆盖以下后端模块的详细设计：

| 模块 | 服务名 | 端口 | Sentinel Agent端口 | 说明 |
|------|--------|------|-------------------|------|
| API网关 | gateway-service | 10010 | 8719 | 路由转发、鉴权拦截、首页聚合、Sentinel网关限流 |
| 公共模块 | tea-common | — | — | 统一返回、异常、JWT工具类、Jackson配置 |
| 用户服务 | user-service | 20001 | 8720 | 注册/登录、个人信息、地址管理、后台用户管理 |
| 内容服务 | content-service | 20004 | 8721 | 文章管理、分类管理、文件上传(MinIO)、后台内容管理 |
| 展览服务 | exhibition-service | 20002 | 8722 | 展览管理、报名管理、后台展览管理 |
| 商品服务 | product-service | 20003 | 8723 | 商品管理、购物车、订单、后台管理 |

### 1.3 设计原则

- **职责分离：** 每个微服务仅负责一个业务域，保持高内聚低耦合
- **无状态设计：** 服务端不存储会话状态，通过JWT Token传递用户上下文
- **防御性设计：** 所有接口校验数据归属和权限，防止越权操作
- **多层容错：** Sentinel 流控熔断 + 重试机制 + onErrorResume 降级 + Redis降级，四层防线保障核心功能可用
- **配置外置：** 数据库连接、Redis地址、JWT秘钥等通过配置文件管理；本地开发使用 application-local.yml 统一管理

## 3. 各模块详细设计

### 3.1 tea-common（公共模块）

#### 3.1.1 核心组件

**Result\<T\> —— 统一响应封装**

| 字段 | 类型 | 说明 |
|------|------|------|
| code | Integer | 状态码（200=成功, 500=通用失败, 自定义码如401/429/503） |
| message | String | 提示消息 |
| data | T (泛型) | 业务数据 |

**静态工厂方法：**

| 方法 | 返回值 | code | 用途 |
|------|--------|------|------|
| `success()` | Result\<Void\> | 200 | 无数据成功响应 |
| `success(T data)` | Result\<T\> | 200 | 携带数据的成功响应 |
| `fail(String message)` | Result\<T\> | 500 | 通用失败响应 |
| `error(int code, String message)` | Result\<T\> | 自定义 | 自定义状态码错误（配合BusinessException使用） |

**BusinessException —— 业务异常**

| 构造函数 | 默认code | 说明 |
|----------|---------|------|
| `BusinessException(String message)` | 500 | 通用业务异常 |
| `BusinessException(Integer code, String message)` | 自定义 | 可指定任意错误码（如401未登录、429限流、503降级） |

继承 `RuntimeException`，`final Integer code` 保证线程安全。

**JwtUtil —— JWT工具类**

| 方法 | 功能 | JWT Payload声明 |
|------|------|-----------------|
| `createToken(userId, username)` | 创建Token（role默认0） | sub, userId, username, role, iat, exp |
| `createToken(userId, username, role)` | 创建Token（指定角色） | 同上 |
| `parseToken(token)` | 解析验证Token，返回Claims | — |

**密钥三级回退机制：** 系统属性 `jwt.secret` → 环境变量 `JWT_SECRET` → 硬编码默认值。生产环境必须设置环境变量覆盖。

**JacksonConfig —— 全局JSON序列化**

三大配置：
1. **Long → String 序列化：** 解决 JavaScript 大数精度丢失问题（Number.MAX_SAFE_INTEGER = 2^53-1）
2. **日期输出格式统一：** LocalDate(`yyyy-MM-dd`)、LocalTime(`HH:mm:ss`)、LocalDateTime(`yyyy-MM-dd HH:mm:ss`)
3. **LocalDateTime 多格式反序列化兼容：** 依次尝试5种输入格式（ISO/T带毫秒/空格带毫秒/空格标准），自动去除尾部Z和时区偏移

#### 3.1.2 模块间引用关系

所有5个业务微服务和gateway-service均依赖 tea-common，通过 `<artifactId>tea-common</artifactId>` 引入。

#### 3.2.2 过滤器链设计

**过滤器执行顺序：**

| 过滤器 | 优先级(Order) | 职责 |
|--------|--------------|------|
| AuthGlobalFilter | `-1` (HIGHEST_PRECEDENCE) | JWT认证 + Redis登录态校验 + 注入用户信息头 |
| SentinelGatewayFilter | `HIGHEST_PRECEDENCE + 1` | 路由级QPS限流 / 熔断降级 |

**设计原则：先认证后限流**——未认证请求不消耗限流配额。

#### 3.2.3 AuthGlobalFilter —— 认证过滤

**公开路径白名单（isPublicPath）：**

| 路径模式 | 方法 | 说明 |
|----------|------|------|
| /user/login | POST | 登录接口 |
| /user/register | POST | 注册接口 |
| /user/test | GET | 测试接口 |
| /user/list | GET | 用户列表（公开浏览） |
| /content/category/** | ANY | 文章分类 |
| /content/article/** | GET | 文章查询（仅读） |
| /content/upload/** | ANY | 文件上传 |
| /exhibition/** | ANY | 展览浏览和报名 |
| /product/** （排除/cart和/order） | ANY | 商品浏览 |
| /home/** | ANY | 首页聚合 |

**注入的请求头（下游微服务直接读取，无需重复解析JWT）：**

| 请求头名 | 来源 | 说明 |
|----------|------|------|
| X-User-Id | JWT.claims.userId | 当前登录用户ID |
| X-Username | JWT.claims.sub | 当前登录用户名 |
| X-Auth-Token | 原始JWT字符串 | 供下游服务做Redis校验 |
| X-User-Role | JWT.claims.role | 当前登录用户角色（默认0） |

**管理员路径校验（handleAdminPath）：**
- `/admin/**` 路径强制要求 `role == 1`（管理员）

**Redis降级策略：**
- 当Redis连接不可用时，打印WARN日志但不中断请求
- 信任JWT本身的签名和有效期，Token仍然可用
- 退出登录时Redis删除失败仅记录日志，不影响响应

#### 3.2.4 HomeAggregationService —— 首页聚合（四层容错架构）

使用 `@LoadBalanced WebClient` 实现响应式微服务调用，通过 `Mono.zip()` 并发请求三个微服务获取文章、展览、商品数据。

**四层容错防线：**

```
第一道防线: Sentinel Gateway Filter（路由级QPS限流）
     ↓ 触发时返回 429 JSON
第二道防线: @SentinelResource（方法级资源保护 + 熔断降级）
     ↓ 各fetch方法独立配置 blockHandler + fallback
第三道防线: WebClient retryWhen（网络瞬时可恢复重试）
     ↓ Retry.backoff(1次, 2s基础间隔, 最大5s退避)，仅重试5xx和网络异常
第四道防线: onErrorResume（最终兜底降级 → 空列表）
     ↓ handleError记录ERROR日志，返回 Collections.emptyList()
```

**Sentinel 资源定义：**

| 资源名 | 业务方法 | blockHandler | fallback |
|--------|---------|--------------|----------|
| fetchArticles | fetchArticles() | fetchArticlesBlockHandler | fetchArticlesFallback |
| fetchExhibitions | fetchExhibitions() | fetchExhibitionsBlockHandler | fetchExhibitionsFallback |
| fetchProducts | fetchProducts() | fetchProductsBlockHandler | fetchProductsFallback |

每个 blockHandler/fallback 行为一致：记录 WARN/ERROR 日志后返回 `Mono.just(Collections.emptyList())`。

**监控指标（AtomicLong计数器）：**

| 指标 | 含义 |
|------|------|
| totalRequests | 总请求数 |
| successCount | 成功数 |

可通过 `getMetrics()` 方法查询。

**数据提取器（函数式接口 DataExtractor）：**
- `extractArticleRecords`：处理 content-service 的分页结构 `{ code:200, data:{ records:[...] } }`
- `extractListData`：处理 exhibition/product-service 的列表结构 `{ code:200, data:[...] }`，支持多种字段名适配

**WebClient 超时配置（WebClientConfig）：**

| 配置项 | 值 | 说明 |
|--------|-----|------|
| 连接超时 | 5000ms (5s) | TCP连接建立超时 |
| 读超时 | 15000ms (15s) | 下游聚合接口最长等待时间 |
| 写超时 | 10000ms (10s) | 请求发送超时 |

**容错设计总则：** 单个下游失败 → 降级为空列表，不影响其他数据；所有服务均失败 → 返回空HomeVO(code=200)，前端展示空状态页。

#### 3.2.5 SentinelGatewayConfig & GatewayBlockHandler ★v2.0新增

**SentinelGatewayConfig —— 网关Sentinel集成配置**

- 在构造阶段注册自定义 `BlockRequestHandler`（避免 @PostInit 延迟初始化问题）
- 将 Sentinel 拦截委托给 `GatewayBlockHandler.handleBlockRequest()` 处理
- `sentinelGatewayFilter()` Bean 优先级设为 `Ordered.HIGHEST_PRECEDENCE + 1`，确保在 AuthGlobalFilter 之后执行

**GatewayBlockHandler —— 限流/熔断统一响应处理器**

| 方法 | 场景 | HTTP状态码 | 返回消息 |
|------|------|-----------|---------|
| handleBlockRequest | BlockRequestHandler接口入口 | 429 | "请求过于频繁，请稍后再试" |
| blockHandler | 限流场景 | 429 | "请求过于频繁，请稍后再试" |
| fallbackHandler | 熔断场景 | 503 | "服务暂时不可用，请稍后再试" |

技术细节：
- static ObjectMapper 单例，避免重复创建降低 GC
- Content-Type 显式指定 `charset=UTF-8` 解决中文乱码
- JSON 序列化失败时返回硬编码 UTF-8 字节数组作为兜底
- 统一返回 `{"code": 429/503, "message": "...", "data": null}` 格式

#### 3.3.4 业务逻辑要点

**注册流程（/user/register）：**
1. 校验用户名是否已存在（username唯一索引）
2. BCryptPasswordEncoder 加密密码
3. 插入用户记录（role默认0，status默认1）
4. 异常时抛出 BusinessException

**登录流程（/user/login）：**
1. 根据username查询用户
2. BCryptPasswordEncoder 校验密码
3. JwtUtil.createToken(userId, username, role) 签发JWT
4. Redis缓存Token：key=`login:{token}`, value=userId, TTL=24h
5. 返回JWT字符串

**修改密码（/user/password）：**
1. 从请求头X-User-Id获取当前用户ID
2. 查询用户，BCrypt校验旧密码
3. BCrypt加密新密码并更新

**收货地址管理要点：**
- 所有写操作从请求头 X-User-Id 获取用户ID（网关注入）
- 未登录（无X-User-Id头）抛出 BusinessException(401)
- 所有写操作校验 address.getUserId().equals(userId) 归属权
- 默认地址互斥：设置任一地址为默认时先批量清除该用户其他默认标记

#### 3.6.4 业务逻辑设计

**商品上架判断逻辑（关键SQL设计）：**
- C端列表/详情查询条件：`status = 1 OR status IS NULL`（修复MySQL中 `status <> 0` 无法匹配NULL行的问题）
- 管理端不受此限制，可查看/管理所有状态商品

**购物车逻辑：**

1. **加入购物车（POST /product/cart/add）：**
   - 从请求头 X-User-Id 获取用户身份（未登录返回401）
   - 校验商品存在性、上架状态(status=1)、库存充足(quantity <= stock)
   - 已存在的购物车记录采用**累加策略**（安全处理null值）
   - 插入或更新购物车记录

2. **购物车列表（GET /product/cart/list）：**
   - 查询当前用户的全部购物车记录
   - 通过 selectBatchIds 批量查询关联商品信息
   - 组装 CartVO（含 subtotal = price × quantity 计算字段）
   - 商品已删除时有降级处理（显示"商品已下架"）

3. **删除购物车（DELETE /product/cart/{id}）：**
   - 校验购物车记录归属权（cart.userId == currentUserId）
   - 防止越权删除他人购物车

4. **修改购物车数量（PUT /product/cart/update）★v2.0新增：**
   - 校验商品存在性和库存充足性
   - 更新指定购物车项的数量

**订单逻辑：**

1. **单商品下单（POST /product/order/create）：**
   ```
   校验商品存在 → 校验库存充足 → 创建Order → 创建OrderItem → 扣减库存
   ```
   全部在 `@Transactional(rollbackFor = Exception.class)` 事务中完成。
   - 订单号生成算法：UUID去掉横杠后取前16位
   - 默认订单状态设为1（已支付）
   - 所有数值字段均有 null 安全处理

2. **购物车批量下单（POST /product/order/create-from-cart）：**
   - 遍历购物车每项创建对应的 OrderItem
   - 逐个扣减各商品的库存
   - 删除购物车记录
   - 总金额 = Σ(price × quantity)
   - 同样在事务中完成

## 5. Sentinel 流控熔断设计（v2.0 新增）

### 5.1 整体架构

Sentinel 作为系统的第二道防线（在网关认证之后），覆盖两个层面：

```
请求进入
    ↓
① AuthGlobalFilter (JWT认证) ← 第一道：身份验证
    ↓
② SentinelGatewayFilter     ← 第二道：路由级QPS限流（网关层面）
    ↓
③ @SentinelResource         ← 第三道：方法级流控/熔断（应用层面）
    ↓
业务逻辑执行
```

### 5.2 网关层 Sentinel 配置

**配置位置：** gateway-service 的 `SentinelGatewayConfig.java` + `application.yml`

| 配置项 | 值 | 说明 |
|--------|-----|------|
| sentinel.transport.dashboard | localhost:8089 | Dashboard控制台地址 |
| sentinel.transport.port | 8719 | Agent与Dashboard通信端口 |
| sentinel.eager | true | 启动时立即建立连接（饥饿加载） |
| sentinel.log.dir | ./logs/sentinel | 日志目录 |
| sentinel.scg.fallback.mode | response | 降级模式：直接返回响应体 |
| sentinel.scg.fallback.response-status | 429 | 降级HTTP状态码 |
| sentinel.scg.fallback.response-body | {"code":429,"message":"请求过于频繁","data":null} | 降级响应体 |

**GatewayBlockHandler 统一响应格式：**

```json
// 限流场景 (HTTP 429)
{"code": 429, "message": "请求过于频繁，请稍后再试", "data": null}

// 熔断场景 (HTTP 503)
{"code": 503, "message": "服务暂时不可用，请稍后再试", "data": null}
```

### 5.3 各微服务 Sentinel 配置

每个微服务的 Sentinel Agent 使用独立通信端口，避免冲突：

| 服务 | Dashboard地址 | Agent端口 | eager | 保护资源 |
|------|--------------|----------|-------|---------|
| gateway-service | localhost:8089 | 8719 | true | 所有路由 |
| user-service | localhost:8089 | 8720 | true | userLogin, userRegister, userUpdatePassword |
| content-service | localhost:8089 | 8721 | true | fileUpload |
| exhibition-service | localhost:8089 | 8722 | true | exhibitionSignup |
| product-service | localhost:8089 | 8723 | true | orderCreate, orderCreateFromCart |

### 5.4 @SentinelResource 保护资源清单

| 服务 | 资源名 | 对应方法 | 保护目标 |
|------|--------|---------|---------|
| user-service | userLogin | UserController.login() | 防止暴力破解/撞库 |
| user-service | userRegister | UserController.register() | 防恶意注册刷接口 |

### 5.5 全局异常处理中的 Sentinel 集成

每个微服务的 `GlobalExceptionHandler` 统一处理以下 Sentinel 异常：

| 异常类型 | HTTP状态码 | 返回消息 | 触发场景 |
|----------|-----------|---------|---------|
| FlowException | 429 | "系统繁忙，请稍后再试" | QPS超阈值限流 |
| DegradeException | 503 | "服务繁忙，已自动降级" | 慢调用比例熔断/异常比例熔断 |

| 其他 BlockException | 默认fail | "请求被限制，请稍后再试" | 未分类的Sentinel拦截 |

此外，user-service 的 GlobalExceptionHandler 还额外处理 JWT 相关异常：

| 异常类型 | 返回消息 |
|----------|---------|
| ExpiredJwtException | "token已过期，请重新登录" |
| MalformedJwtException / SignatureException | "token无效" |

## 6. 安全设计

### 6.1 认证流程

```
┌──────────┐         ┌──────────┐         ┌──────────┐
│  Client  │         │ Gateway  │         │  Redis   │
└────┬─────┘         └────┬─────┘         └────┬─────┘
     │  登录请求           │                    │
     │ ──────────────────►│                    │
     │                    │ 转发到user-service  │
     │                    │ 签发JWT             │
     │                    │ ───────────────────►│ 缓存Token
     │                    │   key=login:{token} │   value=userId
     │  返回JWT Token     │   TTL=24h           │
     │ ◄──────────────────│                    │
     │                    │                    │
     │  业务请求+Token    │                    │
     │ ──────────────────►│                    │
     │                    │ 解析JWT             │
     │                    │ ───────────────────►│ 校验Token存在
     │                    │ ◄───────────────────│
     │                    │ 注入用户信息头       │
     │                    │ Sentinel限流检查     │
     │                    │ 转发到下游微服务     │
     │  业务响应           │                    │
     │ ◄──────────────────│                    │
```

### 6.2 安全措施清单

| 层次 | 措施 | 实现 |
|------|------|------|
| 密码安全 | BCrypt哈希存储 | BCryptPasswordEncoder |
| 传输安全 | JWT签名(HMAC-SHA256) | JwtUtil.createToken/parseToken |
| 会话管理 | Token有效期24小时 + Redis状态可撤销 | login:{token} → userId, TTL=24h |
| 流控防护 | Sentinel 三层限流 | 网关QPS限流 + 方法级@SentinelResource + 热点参数限流 |
| 熔断降级 | 慢调用比例/异常比例自动熔断 | Sentinel DegradeException → 503 |
| 权限控制 | 网关层角色校验(role=1) | AuthGlobalFilter.handleAdminPath |
| 水平越权防护 | 地址/购物车/订单操作前校验userId归属 | 各Controller中的归属校验 |
| 垂直越权防护 | 管理员不可禁用/删除其他管理员 | AdminUserController中的role校验 |
| 信息泄露防护 | 用户查询强制清除password | u.setPassword(null) |
| 密钥管理 | JWT密钥三级回退 | 系统属性 > 环境变量 > 硬编码默认值 |
| Redis容错 | Redis不可用时信任JWT本身 | catch异常后仅记录WARN日志 |
| 参数校验 | Jakarta Validation | @Valid + @NotBlank/@NotNull/@Min/@Length |

## 7. 异常处理设计

### 7.1 全局异常拦截器

每个微服务配置 `GlobalExceptionHandler`（`@RestControllerAdvice`），统一捕获并转换为 `Result<T>` 格式响应。

### 7.2 异常分类与处理策略

| 优先级 | 异常类型 | HTTP状态码(code) | 返回消息示例 | 处理方式 |
|--------|----------|-----------------|-------------|---------|
| 1 | FlowException | 429 | "系统繁忙，请稍后再试" | Sentinel限流拦截 |
| 2 | DegradeException | 503 | "服务繁忙，已自动降级" | Sentinel熔断降级 |
| 3 | ParamFlowException | 429 | "操作过于频繁，请稍后再试" | Sentinel热点限流 |
| 4 | 其他BlockException | 默认fail | "请求被限制" | Sentinel通用拦截 |
| 5 | ExpiredJwtException | 默认 | "token已过期，请重新登录" | 仅user-service |
| 6 | MalformedJwtException | 默认 | "token无效" | 仅user-service |
| 7 | BusinessException | e.getCode() 或 500 | e.getMessage() | 业务异常（自定义code） |
| 8 | MethodArgumentNotValidException | 默认 | 第一个字段的校验错误信息 | 参数校验失败 |
| 9 | Exception | 默认 | "系统异常，请稍后重试" | 兜底（打印堆栈日志） |

*注：当前所有异常HTTP状态码统一返回200（通过Result.code区分业务状态），前端通过Result.code判断。*

### 7.3 错误码规范

| code范围 | 含义 | 示例 |
|----------|------|------|
| 200 | 成功 | 操作成功 |
| 401 | 未认证/Token失效 | "未登录"、"token已过期" |
| 429 | 请求过于频繁（限流） | "登录请求过于频繁" |
| 500 | 通用业务错误 | "用户不存在"、"密码错误" |
| 503 | 服务降级/暂不可用 | "登录服务暂不可用" |
