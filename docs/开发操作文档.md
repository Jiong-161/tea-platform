## 1. 项目概述

**茶文·茶文化品鉴购销服务平台** 是一个集茶文化内容传播、茶事展览活动、茶叶电商交易于一体的垂直领域综合型 Web 平台。采用前后端分离架构，后端基于 Spring Cloud Alibaba 微服务体系，前端包含面向消费者的 C 端前台和面向管理员的后台管理系统。

### 技术栈总览

| 层级       | 技术选型                                | 版本                    |
|----------|-------------------------------------|-----------------------|
| 后端框架     | Spring Boot                         | 3.2.4                 |
| 微服务      | Spring Cloud + Spring Cloud Alibaba | 2023.0.0 / 2023.0.1.0 |
| ORM      | MyBatis-Plus                        | 3.5.7                 |
| 数据库      | MySQL                               | 8.0+                  |
| 缓存       | Redis                               | 6.0+                  |
| 注册中心     | Nacos                               | 2.x                   |
| 对象存储     | RustFS                              | RELEASE.2024-xx       |
| 安全       | JJWT + BCrypt                       | 0.11.5                |
| JDK      | OpenJDK / Oracle JDK                | 17                    |
| 前端框架     | Vue                                 | 3.5+                  |
| 构建工具     | Vite                                | 8.x                   |
| UI 组件库   | Element Plus                        | 2.14+                 |
| 状态管理     | Pinia                               | 3.x                   |
| HTTP 客户端 | Axios                               | 1.x                   |
| 路由       | Vue Router                          | 5.x                   |

##  本地开发环境搭建
### 4.1 安装基础软件
#### 安装 JDK 17
#### 安装 Maven
#### 安装 Node.js （20+）
1. 验证安装
   ```bash
   node -v
   npm -v
   ```

####  安装 Redis
1. 启动方式：脚本启动start-redis.bat
 ```bash
    #Windows Redis 启动批处理脚本
  @echo off
  chcp 65001
  title Redis服务
  echo 正在启动 Redis 服务...
  redis-server.exe redis.windows.conf
  pause
   ```
2.该.bat文件必须放在 Redis 根目录，和redis-server.exe、redis.windows.conf同级，否则执行时提示找不到程序 / 配置文件。
####  安装 Nacos
1. 下载 Nacos：
2. 解压后以单机模式启动：
   ```
   # Windows
   进入 bin 目录
    双击 / 执行命令启动（单机模式）
    方式 1（推荐，直接双击）：
    双击 startup.cmd
    方式 2（命令行，防止报错）：
    在 bin 目录 按住 Shift + 右键 → 在此处打开命令窗口，输入：
    cmd
    startup.cmd -m standalone
   ```
3. 访问控制台：http://localhost:8848/nacos （默认账号密码：nacos/nacos）

#### 安装 RustFS

1. 下载 RustFS
2. 启动 RustFS：
    
    ```
   双击rustfs-launcher.exe
   ```
1. 填写data port host accesskey secretkey   
   
3. 访问控制台：http://localhost:9001 （账号：admin / 12345678）
4. 创建 Bucket：在控制台中创建名为 `tea-platform` 的 Bucket，并设置访问策略为公开读

### 4.3 启动后端服务

| 4 | exhibition-service | IDEA 运行 `ExhibitionApplication.main()` | 20002 |
| 5 | product-service | IDEA 运行 `ProductApplication.main()` | 20003 |

> **提示**：网关服务建议第一个启动，其他业务服务启动顺序无严格限制。

**第四步：验证服务注册**

访问 Nacos 控制台 http://localhost:8848/nacos，在"服务列表"中应看到 5 个已注册服务。

**第五步：验证网关路由**

```bash
curl http://localhost:10010/home/index
# 应返回 JSON 格式的首页聚合数据
```

## 5. 编码规范

### 5.1 命名约定

#### 5.1.1 Java 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 类名 | UpperCamelCase | `UserController`, `ArticleService` |
| 方法名 | lowerCamelCase | `getUserById()`, `createOrder()` |
| 变量名 | lowerCamelCase | `userId`, `orderList` |
| 常量名 | UPPER_SNAKE_CASE | `MAX_PAGE_SIZE`, `DEFAULT_STATUS` |
| 包名 | 全小写 | `com.tea.user.controller` |
| 接口 | UpperCamelCase，不使用 I 前缀 | `UserService` |
| 实现类 | 接口名 + Impl | `UserServiceImpl` |
| Mapper | 实体名 + Mapper | `UserMapper` |
| DTO | 功能描述 + DTO | `LoginDTO`, `UserQueryDTO` |

#### 5.1.2 数据库命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 表名 | tea_ 前缀 + 小写下划线 | `tea_user`, `tea_order_item` |
| 字段名 | 小写下划线 | `user_id`, `created_time` |
| 索引名 | idx_ 前缀 + 字段名 | `idx_username`, `idx_user_id` |
| 唯一索引 | uk_ 前缀 + 字段名 | `uk_order_no` |

#### 5.1.3 前端命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 组件文件名 | PascalCase | `ArticleList.vue`, `ProductDetail.vue` |
| JS/TS 文件名 | camelCase 或 kebab-case | `request.js`, `urlCleaner.js` |
| 路由 name | PascalCase | `ArticleList`, `ProductDetail` |
| 路由 path | kebab-case | `/my-order`, `/article/:id` |
| 变量/函数 | lowerCamelCase | `userInfo`, `fetchData()` |
| 常量 | UPPER_SNAKE_CASE | `VITE_API_URL` |
| Pinia Store | camelCase | `useUserStore`, `useCartStore` |
| CSS 类名 | BEM 或 kebab-case | `header-nav`, `article-list__item` |

### 5.2 代码风格

#### 5.2.1 Java 代码风格

- 缩进：4 个空格
- 字符编码：UTF-8
- 行宽：不超过 120 字符
- 大括号：同行不换行（K&R 风格）
- 使用 Lombok 简化 POJO（`@Data`, `@Builder`, `@NoArgsConstructor`, `@AllArgsConstructor`）
- Controller 方法返回统一 `Result<T>` 响应
- Service 层接口与实现分离
- 优先使用 SLF4J 日志（`log.info()`, `log.error()`），禁止使用 `e.printStackTrace()`

**Controller 示例：**

```java
@RestController
@RequestMapping("/user")
@Slf4j
public class UserController {

    @Autowired
    private UserService userService;

    @PostMapping("/login")
    public Result<String> login(@RequestBody LoginDTO loginDTO) {
        String token = userService.login(loginDTO);
        return Result.success(token);
    }
}
```

**Service 示例：**

```java
public interface UserService {
    String login(LoginDTO loginDTO);
}

@Service
@Slf4j
public class UserServiceImpl implements UserService {

    @Autowired
    private UserMapper userMapper;

    @Override
    public String login(LoginDTO loginDTO) {
        // 业务逻辑
    }
}
```

#### 5.2.2 Vue 代码风格

- 缩进：2 个空格
- 组件结构顺序：`<script setup>` → `<template>` → `<style scoped>`
- 使用 `<script setup>` 语法糖
- 使用 Composition API（`ref`, `computed`, `onMounted`）
- 模板中使用简写：`v-if` / `v-for` / `@click` / `:prop`
- 样式使用 `<style scoped>` 避免全局污染

**Vue 组件示例：**

```vue
<script setup>
import { ref, onMounted } from 'vue'
import { getArticleList } from '@/api/content'

const articles = ref([])
const loading = ref(false)

onMounted(async () => {
  loading.value = true
  try {
    const res = await getArticleList()
    articles.value = res.data.records
  } finally {
    loading.value = false
  }
})
</script>

<template>
  <div class="article-list">
    <el-card v-for="item in articles" :key="item.id">
      {{ item.title }}
    </el-card>
  </div>
</template>

<style scoped>
.article-list {
  padding: 20px;
}
</style>
```

### 5.3 注释要求

#### 5.3.1 Java 注释

- 类注释：说明类的职责和功能
- 公共方法注释：说明方法用途、参数含义、返回值
- 复杂业务逻辑：行内注释说明关键步骤
- 禁止无意义注释或注释掉的代码

```java
/**
 * 用户服务实现类
 * 处理用户注册、登录、信息管理等业务逻辑
 */
@Service
@Slf4j
public class UserServiceImpl implements UserService {

    /**
     * 用户登录
     * @param loginDTO 登录请求参数（username + password）
     * @return JWT Token 字符串
     * @throws BusinessException 用户名或密码错误时抛出
     */
    @Override
    public String login(LoginDTO loginDTO) {
        // 1. 根据用户名查询用户
        // 2. 校验密码（BCrypt 比对）
        // 3. 生成 JWT Token 并缓存到 Redis
    }
}
```

#### 5.3.2 Vue 注释

- 组件文件顶部：简要说明组件用途
- 复杂计算属性/方法：行内注释
- API 函数：注释接口用途

```javascript
/**
 * 获取文章分页列表
 * @param {Object} params - 查询参数 { page, size, categoryId, keyword }
 * @returns {Promise} 文章分页数据
 */
export function getArticleList(params) {
  return request.get('/content/article/list', { params })
}
```

```
        ├── fix/xxx (修复分支)
        └── refactor/xxx (重构分支)
```

| 分支类型 | 命名规范 | 说明 | 生命周期 |
|---------|---------|------|---------|
| `main` | main | 生产环境代码，保护分支 | 永久 |
| `develop` | develop | 开发集成分支 | 永久 |
| `feature/*` | feature/user-address | 新功能开发 | 合并后删除 |
| `fix/*` | fix/register-500 | Bug 修复 | 合并后删除 |
| `refactor/*` | refactor/api-response | 代码重构 | 合并后删除 |

### 7.2 提交规范

提交信息格式：`<type>(<scope>): <subject>`

**type 类型：**

| 类型 | 说明 |
|------|------|
| feat | 新功能 |
| fix | Bug 修复 |
| docs | 文档更新 |
| style | 代码格式调整（不影响逻辑） |
| refactor | 代码重构 |
| test | 测试相关 |
| chore | 构建/工具变更 |

**scope 范围：**

| 范围 | 说明 |
|------|------|
| user | 用户模块 |
| content | 内容模块 |
| exhibition | 展览模块 |
| product | 商品模块 |
| gateway | 网关模块 |
| common | 公共模块 |
| client | C 端前端 |
| admin | 管理端前端 |

**示例：**

```bash
feat(user): 新增收货地址管理功能
fix(product): 修复购物车数量累加逻辑错误
docs(readme): 更新项目部署文档
refactor(gateway): 统一异常处理机制
```

### 7.3 代码审查流程

1. 开发者在功能分支完成开发后，提交 Pull Request 到 `develop` 分支
2. PR 描述需包含：修改内容、影响范围、测试情况
3. 至少一名团队成员进行 Code Review
4. Review 重点：
   - 代码逻辑正确性
   - 是否符合编码规范
   - 是否存在安全隐患（SQL 注入、XSS 等）
   - 异常处理是否完善
   - 是否影响现有功能
5. Review 通过后合并，如有问题则打回修改

### 7.4 常用 Git 操作

``` text
  创建功能分支
    git checkout develop #切换本地分支到 develop 开发主分支
    git pull origin develop 拉取远程origin仓库 develop 分支最新代码，并合并到本地 develop
    git checkout -b feature/user-address 创建并切换到新分支 feature/user-address

    # 同步 develop 最新代码
    git checkout develop
```

## 8. 开发流程

### 8.1 需求开发流程

```
需求分析 → 技术方案设计 → 编码实现 → 自测 → 代码审查 → 联调测试 → 提交上线
```

### 8.2 后端开发流程

1. **需求分析**：明确接口入参、出参、业务规则
2. **数据库设计**：新增/修改表结构，编写 DDL 脚本
3. **编码实现**：
   - Entity 层：定义数据库实体（`@TableName`, `@TableId`）
   - Mapper 层：继承 `BaseMapper<T>`
   - Service 层：定义接口 + 实现类（继承 `IService<T>`）
   - Controller 层：定义 REST 接口（返回 `Result<T>`）
4. **自测**：使用 Swagger UI 或 Postman 验证接口
5. **网关配置**：如需新路由，在 gateway-service 的 `application.yml` 中添加路由规则

### 8.3 前端开发流程

1. **需求分析**：明确页面布局、交互逻辑、数据来源
2. **页面开发**：
   - 在 `views/` 下创建页面组件
   - 在 `api/` 下封装接口调用
   - 在 `router/index.js` 中注册路由
   - 如需全局状态，在 `stores/` 下创建 Pinia Store
3. **联调测试**：启动前后端，验证数据流转
4. **样式调整**：遵循项目 UI 设计规范

### 8.4 新增微服务流程

1. 在 `tea-backend/` 下创建新模块目录
2. 在父 `pom.xml` 的 `<modules>` 中添加新模块
3. 创建 `pom.xml`，引入必要依赖
4. 编写启动类（`@SpringBootApplication(scanBasePackages = "com.tea")`）
5. 编写 `bootstrap.yml`（配置 Nacos 地址和服务名）
6. 编写 `application.yml`（配置端口、数据库等）
7. 在 gateway-service 的 `application.yml` 中添加路由规则
8. 启动并验证服务注册到 Nacos

## 13. 常见问题排查与解决

### 13.1 环境搭建类

#### Q1：Maven 依赖下载失败

**现象**：`mvn clean install` 报错，无法下载依赖

**排查步骤**：
1. 检查网络连接
2. 确认 Maven 镜像配置（推荐使用阿里云镜像）
3. 清理本地仓库缓存：删除 `~/.m2/repository` 中对应依赖目录
4. 强制更新：`mvn clean install -U`

#### Q2：npm install 失败

**现象**：前端依赖安装报错

**排查步骤**：
1. 切换 npm 镜像：`npm config set registry https://registry.npmmirror.com`
2. 清除缓存：`npm cache clean --force`
3. 删除 `node_modules` 和 `package-lock.json` 后重新安装
4. 确认 Node.js 版本 >= 18

#### Q3：Nacos 启动失败

**现象**：Nacos 无法启动或服务注册失败

**排查步骤**：
1. 确认 JDK 版本为 17（Nacos 2.x 要求）
2. 确认使用单机模式启动：`startup.cmd -m standalone`
3. 检查 8848 端口是否被占用：`netstat -ano | findstr :8848`
4. 检查 Nacos 日志：`nacos/logs/start.out`

### 13.2 后端服务类

#### Q4：微服务启动后未注册到 Nacos

**现象**：服务启动成功但 Nacos 控制台看不到服务

**排查步骤**：
1. 确认 Nacos 已启动且可访问 http://localhost:8848/nacos
2. 检查 `bootstrap.yml` 中 `spring.cloud.nacos.discovery.server-addr` 配置
3. 确认 `pom.xml` 中引入了 `spring-cloud-starter-alibaba-nacos-discovery`
4. 确认引入了 `spring-cloud-starter-bootstrap`（Spring Cloud 2020+ 需要）
5. 查看服务启动日志中是否有 Nacos 注册相关错误

#### Q5：网关路由 404

**现象**：通过网关访问接口返回 404

**排查步骤**：
1. 确认目标服务已启动并注册到 Nacos
2. 检查 gateway-service 的 `application.yml` 路由配置
3. 确认路由谓词路径与请求路径匹配
4. 确认目标服务的 Controller `@RequestMapping` 路径与网关路由一致
5. 检查网关日志：`log.info` 或控制台输出

#### Q6：接口返回 401 Unauthorized

**现象**：已登录但仍返回 401

**排查步骤**：
1. 检查请求头是否携带 `Authorization: Bearer <token>`
2. 检查 Token 是否过期（默认 24 小时）
3. 检查 Redis 中是否存在对应 Token（key: `login:<token>`）
4. 如 Redis 未启动，确认网关已做 Redis 降级处理
5. 检查网关鉴权过滤器的公开路径白名单

#### Q7：注册接口 500 错误

**现象**：用户注册时后端返回 500

**原因**：`UserController.register()` 未给 `role` 字段赋默认值，MyBatis-Plus 插入 NULL 到 NOT NULL 列

**解决方案**：在注册逻辑中添加 `user.setRole(0)`

#### Q8：数据库连接失败

**现象**：服务启动报数据库连接错误

**排查步骤**：
1. 确认 MySQL 已启动：`mysql -u root -proot -e "SELECT 1"`
2. 检查 `application.yml` 中数据库连接配置（URL、用户名、密码）
3. 确认数据库 `tea_platform` 已创建
4. 确认 MySQL 驱动版本与 MySQL 版本匹配
5. 检查 `serverTimezone=Asia/Shanghai` 配置

#### Q9：MinIO 文件上传失败

**现象**：图片上传接口报错

**排查步骤**：
1. 确认 MinIO 已启动：访问 http://localhost:9001
2. 确认 Bucket `tea-platform` 已创建
3. 确认 Bucket 访问策略设置为公开读
4. 检查 `application.yml` 中 MinIO 配置（endpoint、access-key、secret-key）
5. 检查文件大小是否超过 10MB 限制

### 13.3 前端开发类

#### Q10：前端页面刷新 500 错误

**现象**：SPA 页面 F5 刷新后显示 500

**原因**：Vite 代理将浏览器页面请求误转发到后端

**解决方案**：项目已内置 `spa-history-fallback` 中间件处理此问题。如仍出现，检查 `vite.config.js` 中间件是否生效。

#### Q11：前端 API 请求跨域错误

**现象**：浏览器控制台报 CORS 错误

**排查步骤**：
1. 开发环境：确认 `vite.config.js` 中 proxy 配置正确
2. 生产环境：确认 Nginx 或网关已配置 CORS
3. 确认请求路径以 `/user`、`/content`、`/exhibition`、`/product`、`/home`、`/admin` 开头

#### Q12：Token 过期后页面异常

**现象**：Token 过期后页面白屏或数据加载失败

**排查步骤**：
1. 检查 `request.js` 响应拦截器是否正确处理 401 状态码
2. 确认 401 时清除 Token 并跳转登录页
3. 检查路由守卫是否正确拦截未登录用户

#### Q13：Long 类型 ID 精度丢失

**现象**：后端返回的 ID 在前端显示不正确（如 `9007199254740993` 变成 `9007199254740992`）

**原因**：JavaScript Number 类型最大安全整数为 `2^53 - 1`，超过此范围的 Long 值会丢失精度

**解决方案**：项目已集成 `json-bigint`（`storeAsString: true`），自动将大数转为字符串处理。如仍有问题，检查 `request.js` 中 `transformResponse` 配置。

#### Q14：图片 URL 空格导致 404

**现象**：图片加载失败，URL 中包含 `%20`

**原因**：数据库中图片 URL 包含空格字符

**解决方案**：项目已集成 `deepCleanUrls` 工具自动清洗 URL 中的空格。如仍有问题，检查数据库中的 URL 数据。

### 13.4 数据库类

#### Q15：初始用户无法登录

**现象**：使用 SQL 脚本初始化的账号无法登录

**原因**：SQL 脚本中密码为明文，但登录使用 BCrypt 验证

**解决方案**：使用 BCrypt 哈希值替换明文密码。明文 `123456` 对应的 BCrypt 哈希值可通过以下方式获取：

```java
import org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder;
new BCCryptPasswordEncoder().encode("123456")
```

或直接使用已修复的 SQL 脚本。

#### Q16：数据库表不存在

**现象**：服务启动报 Table doesn't exist

**排查步骤**：
1. 确认已执行 DDL 建表脚本
2. 确认连接的数据库为 `tea_platform`
3. 检查 `application.yml` 中数据库 URL 配置
4. 确认 MyBatis-Plus 表名前缀配置与实际表名一致

### 13.5 Redis 类

#### Q17：Redis 未启动导致鉴权失败

**现象**：未安装 Redis 时，登录后仍被拦截

**解决方案**：项目已对 Redis 调用进行 try-catch 降级处理，Redis 不可用时信任 JWT 本身签名和有效期。如仍有问题，检查 `AuthGlobalFilter.java` 中降级逻辑。

#### Q18：Redis 连接超时

**现象**：服务日志报 Redis 连接超时

**排查步骤**：
1. 确认 Redis 已启动：`redis-cli ping`（应返回 `PONG`）
2. 检查 `application.yml` 中 Redis 地址和端口配置
3. 如有密码，确认密码配置正确
4. 检查防火墙是否放行 6379 端口
